Step 01 · Where Everything Begins

Problem Statement

Read this first. This is the only input. Everything else in this document is derived from it.

📌 What is a Problem Statement?

A problem statement is a plain-language description of a real-world situation that a system needs to solve. It is written by the client or business owner — not a developer. It contains no technical terms, no database concepts, no class names. It simply describes what the business does and what problems it faces.

Your job as a developer is to read it carefully, understand the domain, and then translate it into structured requirements (SRS), then into design artifacts (ERD, Class Diagram), and finally into code.

The Original Problem Statement

A restaurant chain wants to develop a system to manage its operations, including branches, staff, customers, menus, and orders. The chain operates several branches, each with a unique branch ID, name, address, contact number, and opening hours. Each branch is managed by a branch manager and has several employees such as chefs, waiters, and cashiers.

Every employee has an employee ID, full name, position, salary, date of hire, and contact information. Employees are assigned to specific branches, and some may work shifts across multiple branches when needed. Each branch also maintains shift schedules, recording which employees worked on specific dates and time slots.

The restaurant offers a wide variety of menu items, each identified by a unique item ID, name, description, price, and category (e.g., appetizer, main course, dessert, beverage). Some items may have optional add-ons or variations, such as extra cheese or large size. Menu items can appear in different branches, and each branch may have its own price adjustments or availability rules based on local preferences.

Customers can dine in, order takeaway, or request delivery. Each customer has a customer ID, full name, phone number, email, and loyalty points balance. Customers may visit multiple branches and place several orders over time.

An order is identified by an order ID and includes order type (dine-in, takeaway, delivery), date and time, total amount, payment method (cash, card, wallet), and status (pending, preparing, served, completed, canceled). Orders are linked to a specific branch and handled by a responsible staff member. Each order contains one or more order items, representing the menu items purchased. For each order item, details such as quantity, unit price, special notes, and any selected add-ons should be stored.

The restaurant also keeps records of inventory, where each branch maintains its stock of ingredients. Each ingredient has an ingredient ID, name, unit (kg, liter, piece), and current quantity. Menu items are prepared using multiple ingredients, and each ingredient can be used in several menu items. The recipe relationship between menu items and ingredients should include the required quantity of each ingredient per serving.

For deliveries, the system tracks delivery staff, each having a staff ID, vehicle type, license number, and assigned area. Delivery records should include delivery ID, order ID, delivery address, delivery time, and status (on the way, delivered, failed).

Additionally, customers can leave feedback for orders they've completed. Each feedback record includes a feedback ID, date, rating, comments, and references the corresponding customer and order.

💡 How to Read a Problem Statement

When reading, highlight or note: nouns (these will likely become your data entities later), verbs (these will become operations/use cases), and constraints (these will become business rules and validations). But don't jump there yet — the next step is writing the SRS first.

Step 02 · Before Any Design

Software Requirements Specification (SRS)

Translate the problem statement into structured, clear requirements — still no technical design.

📖 What is an SRS?

An SRS (Software Requirements Specification) is a formal, structured document that describes what a software system must do, without specifying how it will do it. Think of it as the contract between the client (who describes the problem) and the development team (who will build the solution).

A good SRS answers: What must the system do? What must the system not do? Who uses it? What are the rules? It is written in clear, numbered requirements so that nothing is ambiguous or forgotten.

Important: The SRS has zero technical details. No tables, no classes, no database, no code. Only what, never how.

✍️ Do Students Write This or Read It?

In a real project, you write the SRS yourself based on the client's problem statement. In an academic exercise, you are often given the problem statement and must write the SRS as your first deliverable. The SRS you see below is what you would produce after reading the problem statement above. All information used later in this document — actors, entities, stories, rules — comes only from this SRS, not from guessing or prior knowledge.

Software Requirements Specification — Restaurant Chain Management System
Version 1.0 Type: Console-Based OOP Application Storage: File-Based (No Database) Audience: Backend Development Students
Chapter 1
System Overview

The Restaurant Chain Management System is a console-based application that allows restaurant staff at one or more branches to manage daily operations. These operations include managing employees and their shifts, maintaining a menu catalogue, processing customer orders (dine-in, takeaway, delivery), tracking inventory, managing deliveries, and collecting customer feedback. The system stores all data in files. There is no graphical interface.

SRS-01
The system shall support managing multiple restaurant branches simultaneously.
SRS-02
The system shall be operated through a text-based console menu by authorized restaurant staff only.
SRS-03
All data shall be stored in and loaded from files. No database management system shall be used.
Chapter 2
Branch Management Requirements
SRS-04
The system shall allow creating, viewing, updating, and removing branch records. Each branch has a unique identifier, name, address, contact number, and opening hours.
SRS-05
Each branch must be associated with exactly one branch manager at all times. A branch cannot exist without a manager.
SRS-06
The system shall prevent registering two branches with the same name and address combination.
Chapter 3
Employee Management Requirements
SRS-07
The system shall allow registering employees with their full name, position, salary, date of hire, and contact information. Employee positions are limited to: Chef, Waiter, Cashier, Branch Manager, Delivery Staff.
SRS-08
Each employee must be assigned to at least one branch. An employee may work shifts at multiple branches.
SRS-09
The system shall maintain shift schedules, recording which employee worked at which branch on a given date and time slot. An employee cannot have two overlapping shifts on the same day.
SRS-10
Employee salary must be a positive number. Date of hire cannot be set to a future date.
SRS-11
Employee contact information (phone and email) must be unique across all employees.
Chapter 4
Menu Management Requirements
SRS-12
The system shall maintain a global catalogue of menu items. Each item has a unique identifier, name, description, base price, and category. Categories are limited to: Appetizer, Main Course, Dessert, Beverage.
SRS-13
Menu items may have optional add-ons or variations (e.g., extra cheese, large size). Each add-on has a name and an additional price of zero or more.
SRS-14
Each branch may override the price of any global menu item for local pricing. Each branch may also mark a menu item as unavailable at that branch.
SRS-15
Menu item base prices must be greater than zero. Item names must be unique in the global catalogue.
Chapter 5
Customer Management Requirements
SRS-16
The system shall allow registering customer profiles with a full name, phone number, email address, and a loyalty points balance starting at zero.
SRS-17
Customer phone numbers and emails must be unique. Phone numbers must follow a valid format. Emails must contain an "@" and a domain.
SRS-18
A customer may place orders at multiple branches over time.
Chapter 6
Order Management Requirements
SRS-19
The system shall allow creating orders of three types: Dine-In, Takeaway, and Delivery. Each order is linked to a branch, a customer, and a handling staff member.
SRS-20
An order must contain at least one item. Each order item records the menu item, quantity (minimum 1), unit price at time of order, optional special notes, and any selected add-ons.
SRS-21
The unit price recorded on an order item must reflect the price at the time of placement, including any branch price override. It must not change if the global price is later updated.
SRS-22
Order status follows a strict lifecycle: Pending → Preparing → Served → Completed. Cancellation is allowed from Pending status by any staff, and from Preparing status by a Branch Manager only.
SRS-23
Payment can only be processed when an order is in Served status. Accepted payment methods are: Cash, Card, and Wallet. After payment, the order is marked Completed and cannot be changed.
SRS-24
Upon successful payment, the system shall automatically add loyalty points to the customer's account at a rate of one point per currency unit spent (rounded down).
Chapter 7
Inventory & Recipe Requirements
SRS-25
Each branch shall maintain its own inventory of ingredients. An ingredient record has a name, unit (kg, liter, or piece), and current quantity. Quantities must be zero or greater.
SRS-26
Each menu item shall have a recipe that lists the ingredients required and the quantity of each ingredient needed per serving.
SRS-27
When an order status changes from Pending to Preparing, the system shall automatically deduct the required ingredient quantities from the branch inventory based on the recipe and the quantity of each item ordered.
SRS-28
If a deduction would cause any ingredient to fall below a defined low-stock threshold, the system shall display a warning alert before confirming the status change.
Chapter 8
Delivery Requirements
SRS-29
Delivery staff profiles shall include a staff ID, vehicle type, license number, and assigned delivery area.
SRS-30
A delivery order must have a non-empty delivery address. The system shall create a delivery record automatically when a delivery-type order is confirmed.
SRS-31
A delivery record tracks: delivery ID, linked order, delivery address, assigned delivery staff, delivery timestamp, and status (On the Way, Delivered, Failed).
SRS-32
A delivery staff member may only be assigned to one active delivery at a time. They become available again when their delivery is marked Delivered or Failed.
SRS-33
If a delivery fails, a reason must be recorded. The linked order shall be flagged for manager review.
Chapter 9
Feedback Requirements
SRS-34
A customer may submit feedback for a completed order. Feedback includes a numeric rating from 1 to 5 and an optional text comment.
SRS-35
Feedback can only be submitted for orders with status Completed. Only one feedback record is allowed per order per customer.
SRS-36
Each feedback record stores: a unique feedback ID, the date submitted, the rating, the comment, and references to the customer and order.
Chapter 10
Architecture & Implementation Constraints
SRS-37
The solution shall be organized into three simple projects: Core for business entities and contracts, Infrastructure for file-based persistence and seed data, and UI for console menus, display formatting, and user interaction.
SRS-38
The design shall prefer interfaces and composition for behavior sharing. Inheritance may be used only when it adds clear value and keeps the model simpler.
SRS-39
Entity creation shall use simple factory methods where they improve readability, validation, and consistency of object creation. Factories must remain lightweight and must not hide important business rules.
SRS-40
The implementation shall follow KISS and the SOLID principles, with clean naming, small focused methods, and explicit validation. Over-engineering and unnecessary patterns are not allowed.
SRS-41
The codebase shall use extension methods and, where appropriate in .NET 10, extension properties only for small reusable helpers that improve readability without hiding business logic.
SRS-42
The implementation shall favor clean, professional, production-style code with clear separation of concerns, predictable control flow, and simple testable components. No code generation or framework-heavy patterns are required.
Step 03 · Extracted from SRS

Scope & Assumptions

What this system will and will not do — and the decisions we made to fill in gaps the SRS left open.

📖 What are Scope and Assumptions?

Scope defines the clear boundary of the system. It answers: what is inside our responsibility and what is not? Every item in the scope should trace back to at least one SRS requirement. Items out of scope exist to prevent "scope creep" — building features no one asked for.

Assumptions fill the gaps. No SRS is perfectly complete. When the SRS leaves something unclear or unspecified, the developer must make a clear, written assumption so that everyone agrees on the behavior. Assumptions are not guesses — they are documented decisions.

Scope
✅ In Scope
→Manage multiple branches (SRS-01, SRS-04)
→Employee registration, assignment, and shift scheduling (SRS-07–SRS-11)
→Global menu catalogue with per-branch pricing & availability (SRS-12–SRS-15)
→Customer registration with loyalty points (SRS-16–SRS-18)
→Orders: dine-in, takeaway, delivery — full lifecycle (SRS-19–SRS-24)
→Branch inventory with auto-deduction on order preparation (SRS-25–SRS-28)
→Delivery staff tracking and delivery record management (SRS-29–SRS-33)
→Customer feedback for completed orders (SRS-34–SRS-36)
→File-based data storage and loading (SRS-03)
→Console menu interface for all operations (SRS-02)
🚫 Out of Scope
✗Any graphical user interface (GUI)
✗Real-time online or mobile ordering
✗Live payment gateway (payment is simulated)
✗GPS-based delivery tracking
✗Automated supplier reordering
✗Payroll calculation or HR management
✗Loyalty point redemption at checkout
✗Multi-language or currency support
✗Network/cloud database
Assumptions — Filling SRS Gaps
A-01
The SRS says employees can work at multiple branches but doesn't define a "primary branch." We assume each employee has one primary branch stored on their profile, and all other branches are accessed through shift records only.
A-02
SRS-27 says inventory is deducted when an order moves to Preparing, but doesn't say what happens if inventory is insufficient. We assume the system shows a warning but allows the manager to override and proceed. The operation is not automatically blocked.
A-03
SRS-28 says "low-stock threshold" but doesn't define it. We assume a default threshold of 5 units per ingredient. Branch managers can update this value per ingredient.
A-04
SRS-24 says loyalty points are awarded at payment, but doesn't say when delivery orders are paid. We assume delivery orders are also paid through the cashier console, after the delivery is confirmed as Delivered.
A-05
The SRS does not specify who manages customers. We assume any branch staff (Waiter, Cashier) can register or look up customer profiles.
A-06
The SRS doesn't mention user login. We assume there is a simple role-based menu system: the user selects their role (Admin, Manager, Waiter, etc.) when starting a session. No real authentication is required for this version.
A-07
The SRS mentions "Delivery Staff" separately from other employees. We assume Delivery Staff is a type of employee with additional delivery-specific attributes (vehicle, license, area) stored separately.
Step 04 · Extracted from SRS

Actors

Who interacts with the system? Every actor traces back to the SRS.

📖 What is an Actor?

An actor is any person or role that interacts with the system from outside. We find actors by asking: "Who is mentioned in the SRS as someone who does something?" Actors are not implemented as code — they represent the humans who will use the system. We identify them before writing user stories because every user story must have an actor.

Step 05 · Derived from SRS + Actors

User Stories

Each story is mapped to its source SRS requirement. Every validation is also traceable.

📖 What is a User Story?

A user story is a short, goal-focused description of something an actor needs the system to do, written from the actor's perspective. The standard format is:
"As a [actor], I want to [action], so that [business value]."

User stories are not technical. They don't mention classes, files, or methods. They capture intent and business value only.

✍️ How Are User Stories Derived from the SRS?

Step 1 — Read each SRS requirement. Ask: does this require an actor to actively do something? If yes → it is a candidate user story.

Step 2 — Assign the correct actor from the Actors table you already built.

Step 3 — Write the story using the template. The "so that" part comes from the business purpose in the SRS chapter title or the surrounding context.

Step 4 — Add validations. These come from the SRS requirement itself (e.g., SRS-10: "salary must be positive") and from your Assumptions table (e.g., A-03: low-stock threshold = 5).

Step 06 · Derived from SRS + Assumptions

Business Rules

Constraints the system must always enforce — every rule points to its SRS source.

📖 What are Business Rules?

Business rules are non-negotiable constraints that govern how the system behaves. They are different from user stories: stories describe what someone wants to do, while business rules describe what the system must enforce regardless of who is asking.

Business rules come from the SRS (explicit constraints like "salary must be positive") and from assumptions (decisions we made to fill SRS gaps). Later, when you write code, each business rule becomes a validation or guard clause in your methods.

📖 What is a Use Case?

A use case describes exactly how an actor achieves their goal through a sequence of steps. Where a user story says "I want to place a dine-in order", the use case shows the full conversation between the actor and the system: every step, every decision point, every alternative path.

Step 08 · From Requirements to Data Model

Entity-Relationship Diagram (ERD)

We now move from business language into data modeling. This is the first technical step.

📖 What is an ERD and Why Do We Need It?

An Entity-Relationship Diagram (ERD) is a visual map of the data the system needs to remember. It shows what "things" (entities) exist, what information each thing holds (attributes), and how things relate to each other (relationships).

We draw the ERD before writing classes because it forces us to think clearly about the data and its relationships without worrying about code yet. It answers: what data do we need, and how is it connected?

⚠️ Important — We Are NOT Using a Database

The ERD is being used here as a thinking and modeling tool only. We will not build any database. All data will be saved to and loaded from plain files (text, CSV, or JSON). The ERD helps us understand the relationships between our data so we can design our classes and file structures correctly.

Think of it this way: an ERD shows the what and how it connects. Files are just the storage format. Classes are how we represent it in code.

🔎 How Are Entities Derived?

Now that we have a complete SRS, we can finally identify entities systematically — not by guessing. We go through each SRS chapter and ask: "What thing does this chapter ask us to store information about?" Each chapter typically produces one or more entities. Every entity we identify must trace back to a requirement in the SRS.

➡️ What Comes Next — The Bridge to Class Diagram

Now that we have the ERD, we can produce the Class Diagram. The translation is direct and mechanical:

Each entity → one class. Each attribute → a field in that class. Each relationship → an association, list, or composition. In this project we will prefer interfaces and composition in most cases, and use inheritance only when it keeps the model clearly simpler. Associative entities like OrderItem, OrderItemAddOn, Recipe, and ShiftSchedule become their own classes with references to the classes they connect. The Class Diagram is documented in the next section of this project.

Step 09 · Incremental Design

Class Diagram Plan

Build the class diagram in small clicks. Each step adds only what is needed next.

📖 How We Will Build It

We will not draw the full class diagram all at once. That usually creates noise and confusion. Instead, we build it incrementally: start with the core entities, then add order details, then add operational support classes, then add interfaces and helper abstractions.

This fits the project goals: keep it simple, prefer composition over inheritance, and make every class earn its place.

🧠 Design Direction

The class diagram will use interfaces and composition in most places. Delivery-specific data will be composed with Employee, not modeled through deep inheritance. Small factory methods may be shown as static creation methods on entities or dedicated lightweight factory classes only where they make creation cleaner.

Step 10 · Analyze Before Coding

OOP Concepts Mapping

A short analysis of which OOP concepts we will use, where we will use them, and why.

📖 Why This Section Exists

This section is not about showing off concepts. It is about making deliberate design choices. We will use only the OOP ideas that genuinely help this project stay clean, simple, and professional.

ConceptWill We Use It?Where?Why?
EncapsulationYesAll entities such as Order, Branch, CustomerTo keep state valid and ensure rules are enforced through methods instead of uncontrolled public mutation.
AbstractionYesApplication services, repositories, console formatting helpersTo separate domain logic from file storage and UI concerns.
InterfacesYesIFileRepository<T>, ISeeder, optional formatting or menu contractsInterfaces give us clean boundaries between Core, Infrastructure, and UI without adding heavy patterns.
CompositionYes, heavilyOrder with items, Branch with inventory, Employee with optional DeliveryProfileThis is the main modeling style for the project because it is simpler, clearer, and more flexible than inheritance in this domain.
InheritanceMinimalOnly if a later need is truly obvious and simplifies the designWe deliberately avoid deep hierarchies to keep the model easy for students to understand and maintain.
PolymorphismLimited but usefulMostly through interfaces, not class hierarchiesWe want behavior substitution where it helps, but not complex inheritance trees.
Factory MethodsYes, simple onlyEntity creation for valid objects with IDs/default valuesThey keep creation readable and centralized without introducing a full factory pattern infrastructure.
Extension MethodsYesValidation helpers, formatting helpers, ID normalization, common file/path helpersThey improve readability for small repeated logic, as long as business rules stay explicit.
Extension Properties (.NET 10)Maybe, but carefullyOnly tiny convenience helpers where they clearly improve readabilityWe will keep their use limited and never hide important logic behind them.
Design Principles
✅ Principles We Will Follow
→KISS: choose the simplest design that satisfies the requirement
→SRP: each class should have one clear responsibility
→OCP/DIP: use interfaces at project boundaries, especially persistence
→Prefer explicit validation and guard clauses over hidden magic
→Keep methods small, names clear, and flows predictable
🚫 Things We Will Avoid
→Deep inheritance hierarchies for staff roles
→Generic frameworks or patterns that students do not need yet
→Huge service classes that know everything about the system
→Extension methods that hide real business decisions
→Over-engineering in the name of being “professional”

Step 12 · Architecture

Project Structure

A clean three-project solution that keeps responsibilities separated without becoming complicated.

✅ Final Architecture Decision

The solution will contain three basic projects only: RestaurantChain.Core, RestaurantChain.Infrastructure, and RestaurantChain.UI. This is enough to be professional while still staying simple for students.

Project 1 · Core
→Contains entities, value-like domain objects, enums, interfaces, and small factory methods
→Contains business rules and validation guards
→Contains service contracts and application-level abstractions if needed
→Must not know anything about files or console formatting
Project 2 · Infrastructure
→Contains file-based repositories and data source implementations
→Contains seed data loaders and starter sample data
→Responsible for reading and writing JSON, CSV, or text files
→Implements Core interfaces without leaking file details into the UI
Project 3 · UI
→Contains menus, prompts, console styling, summaries, and flow control
→Calls application services and displays results to the user
→Contains no persistence logic and no core business rules
→May contain simple console formatting helpers and display-focused extension methods
Reference Flow
→UI depends on Core
→Infrastructure depends on Core
→Core depends on nothing else
→This preserves clean boundaries and supports SOLID without unnecessary complexity
